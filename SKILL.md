# 鉴定式刑法案例研习超级工作流

本 skill 是一份面向所有用户的、可移植的鉴定式刑法案例分析工作流。它将"三阶层犯罪论体系"与"鉴定式写作"两套方法论编排为四段工作流，内置罪名大纲生成引擎（offense-examination-engine）自主完成犯罪构成鉴定式分析。

裁判依据为中华人民共和国《刑法》（2023 修正）及现行司法解释。本 skill 在犯罪论分歧议题上不预设权威，用户可在 `0-cold-start-interview.md §1.5` 配置学派立场，未配置时默认采行为无价值二元论；同一议题在同一报告内不得反复横跳。

## 一、什么时候用本 skill

用户提交刑事案例材料并需要按学术规范输出案例分析报告时——典型场景：

- 法学院学生 / 教师准备刑法案例研习作业或竞赛；
- 检察官 / 律师 / 法务为复杂刑事案件做"内部全要件检视"作为起诉意见书 / 辩护意见的底稿；
- 法律研究者撰写刑法案例研究文章；
- 用户希望系统理解一个真实刑事案件的全部罪名 / 参与形态 / 竞合关系。

**不适用**：起诉书 / 辩护词 / 量刑意见等具体诉讼文书（请使用专门的刑事诉讼文书 skill）；快速合规咨询（请使用法律研究 skill）。

## 二、核心方法论

### 2.1 理论基底

本 skill 采用**以行为无价值二元论为基础的当代三阶层犯罪论体系**，兼采结果无价值论的合理内核。核心理论立场：

- **构成要件符合性**：同时包含客观构成要件与主观构成要件；
- **违法性**：消极检验正当化事由，且要求客观与主观正当化要素兼备；
- **有责性**：消极检验责任阻却事由，含责任能力、违法性认识可能性、期待可能性。

### 2.2 方法论文件清单

| 模块 | 文件 | 内容 |
|------|------|------|
| 内在结构 | `methodology/inner-structure.md` | 三阶层检验逻辑、涵摄四步法（总起句→定义→归入→结论）、构成要件→违法性→有责性逐层递进 |
| 外在结构 | `methodology/examination-sequence.md` | 五步分析法（事实单元划分→参与人列举→罪名检验→竞合处理→全案结论）、罪名检索顺序、参与人分析顺序 |
| 写作风格 | `methodology/gutachten-writing-rules.md` | 鉴定式 / 裁判式句式、术语统一、法条引用规则、脚注规范、学说争议展开格式 |
| 比例原则 | `methodology/proportionality.md` | 论证深度的分层（明显 / 争议 / 核心争议）、"五花肉"原则 |
| 法律解释工具箱 | `methodology/interpretation-toolkit.md` | 统合版：上编为通用方法论（文义/体系/历史/目的位阶、罪刑法定约束、类推禁止、不确定法律概念类型化、风险自查清单）；下编为刑法分则专属 SOP（客观/实质解释论、法益保护导向、注意规定 vs 法律拟制鉴别四要素、用语相对性分级、同位语测试、兜底条款限缩、23 条权威结论库、罪刑均衡反向校验 15 步 SOP） |


### 2.3 罪名大纲生成引擎

位于 `methodology/offense-examination-engine/`，一套从第一性原理自主完成"罪名发现→规范属性识别→构建检视程式→要件拆解→大纲成型"的方法论引擎。

### 2.4 犯罪类型检视程式库

位于 `methodology/crime-type-library/`，为各种犯罪形态提供标准化检视程式骨架。

## 三、工作流（四段）

```
[首次使用] 0-cold-start-interview  →  写入 .user-config.md（KB 注册 + 工具配置 + 偏好）

每次会话（严格遵守"事实—罪名—规范"阀门，内含于 W1/W2）：
  ┌─────────────────── W1（事实层） ───────────────────┐
  1-case-intake         （PDF/Word/图片 → 中性事实笔记 + 设问逐字 + 事实单元划分 + 参与人列举）
  └────────────────────────────────────────────────────────────┘
  ┌─────────────────── W2（规范层起点 · 两闸独立确认） ───────────────────┐
  2a-preselection          （罪名大纲生成引擎 + MCP双层校验 → 候选罪名清单 → 闸一确认）
  2b-outline               （已确认清单展开为三阶层要件大纲 → 闸二确认）
  └────────────────────────────────────────────────────────────────────────┘
  ┌─────────────────── W3（并行写作） ───────────────────┐
  3-drafting                   （≥2 参与人多 subagent 并行；即时MCP校验；法律解释贯穿）
  └────────────────────────────────────────────────────────────┘
  ┌─────────────────── W4（差异核验+输出） ───────────────────┐
  4-verification-and-output    （增量法条校验+大纲覆盖+事实忠实+格式扫描 → 一次性修订 → Word输出）
  └────────────────────────────────────────────────────────────┘
```

每段独立文件位于 `workflow/` 目录下，用户也可单独调用某一段（如已有大纲直接进入 3-drafting）。

**阀门约束**（内含于 W1/W2）：
- W1 输出**只允许**事实层+设问逐字+事实单元划分+参与人列举——禁止条文号、罪名定性、争点预判；
- W2 是法律层的真正起点——条文号、要件解释、检视程式**只能**从此段开始，且每条文须经 MCP 双层校验；
- 跨阀门污染一律视为方法论违例，应回到对应阶段重做。

## 四、依赖与协同

### 4.1 硬依赖（必须）

- **法律检索 MCP 能力槽（必须）** —— 实时校验法条号、条文原文、现行有效性的工具。本 skill 与具体 MCP 产品**解耦**：用户在冷启动访谈阶段选择法律检索服务商（如北大法宝、华宇元典等），agent 自动探测该服务的可用工具并映射到 5 项能力槽（`get_article` / `search_article` / `get_law_list` / `search_case` / `get_case_list`），写入 `.user-config.md`（参见 `workflow/0-cold-start-interview.md` §2.5）。运行时通过占位符 `${LEGAL_SEARCH.<能力名>}` 解析为实际工具调用。**铁律**：返回项 `${timeliness_field}` 不等于 `${timeliness_active_value}` 的一律不得引用；效力层级排序由 `${effectiveness_field}` 字段确定。**未配置时允许降级，但报告中法条引用必须全量打标**（详见 §六-1）。具体调用模板/决策树详见 `methodology/offense-examination-engine/01-charge-discovery.md` §五。
- **罪名大纲生成引擎**（`methodology/offense-examination-engine/`）—— 本 skill 的**思维内核**，一套从第一性原理自主完成"找法→规范属性识别→构建检视程式→要件拆解→大纲成型"的方法论引擎。


### 4.1.1 可选增强（非必须）

- **用户知识库（如有）** —— 用户自建的刑法评注 / 教科书知识库。当此 KB 可用时，在 `2a-preselection` 与 `3-drafting` 阶段**优先参考**其检视程式与学说观点，并与大纲生成引擎互相校验。当此 KB **不可用**时，大纲生成引擎独立完成全部检视程式构建。

### 4.2 降级策略

| 资源 | 正常路径 | 降级行为 |
|---|---|---|
| 法律检索 MCP 能力槽 | 调用 `${LEGAL_SEARCH.get_article}`/`${LEGAL_SEARCH.search_article}` 实时校验，按 `${timeliness_field}` 过滤 | 条文引用全量标注 `（⚠ 未校验）`，报告末尾附"待校验清单" |
| 大纲生成引擎文件 | 读取 methodology/offense-examination-engine/ | 基于模型犯罪论方法论知识构建临时检视程式，标注 `[方法论自主构建]` |

降级后的产物须在报告首页或闸确认时向用户明示降级状态与影响范围。

## 五、法律解释方法的运用：贯穿增强 + 罪刑法定约束

**根本立场**：法律解释方法（文义/体系/历史/目的）是论证的**增强器**，**贯穿写作全程使用**。

**罪刑法定约束**（区别于民法的特殊规则）：
- 文义解释划定**不可逾越的外延边界**——类推不利于被告人的解释在刑法中被严格禁止；
- 目的限缩仅允许有利于被告人时适用；
- 目的解释不得超出法条文字可能的含义范围。

**触发条件**：要件含义为不确定法律概念 / 一般条款（如"情节严重""数额巨大""特别残忍手段"），或本案事实涵摄存在歧义、学说有分歧、司法实践有分歧。

**素材分级**（与民法相同）：

```
1. 用户已注册的评注 / 教科书 KB（脚注 [评注] / [释义]）——最权威
2. 法律检索 MCP（`${LEGAL_SEARCH.search_case}`）+ 联网学术 / 实务搜索（脚注 [案例] / [联网]）——通常都要执行
3. 方法论自助式解释（依工具箱独立完成）——不加脚注，正文展开方法步骤
```

## 六、关键约束（三层）

### 铁律层（违反即报告失败）

1. **法条 MCP 校验铁律**：本 skill 在任何位置出现法条号或条文原文前，必须经法律检索 MCP 实时校验，确认条文号正确、现行有效、内容一致后方可写入。模型对法条的训练记忆**不得**作为引用依据。具体规则：
   - **校验颗粒度**：同一条文在同一份报告中**首次引用必须 MCP 校验**，后续引用可复用该次校验结果；
   - **校验失败处理**：MCP 返回 not_found / 已废止 / 内容差异时采用**软标注**——`[⚠ MCP 校验失败：<原因>，请人工核对]`；
   - **降级策略**：未配置任何法律检索 MCP 时允许降级，但全部法条引用须统一带 `[⚠ 未经 MCP 校验，请人工核对]` 标注；
   - **校验记录留存**：每次会话生成 `verification-log.md`。

2. **方案先确认再开工 · 两闸独立**：完成 `1-case-intake`（W1）后必须用户确认才能进 W2；W2 拆为两道**独立确认门**——`2a-preselection`（候选罪名清单确认 = 闸一）与 `2b-outline`（检视大纲确认 = 闸二）必须分别用 AskUserQuestion 单独拍板，**严禁合并**；闸二拍板后才能进 W3。

3. **阀门约束：事实 → 罪名/规范**（内含于 W1/W2）：
   - **W1**只允许事实层+设问逐字+事实单元划分+参与人列举。**禁止**罪名定性、条文号、争点预判、工作假设。
   - **W2**是法律层的真正起点：调用大纲生成引擎完成罪名预选、首次出现条文号、形成大纲。
   - 跨阀门污染一律视为方法论违例。

### 规范层（标准遵循、偏离须说明理由）

4. **检视程式骨架**：报告正文须严格遵循三阶层检验逻辑与五步分析法构建分析框架，并以 `canonical-report-template.md` 大纲填充骨架为展开基准。
5. **完整自含的脚注（真实来源原则）**：禁止"参前注 X"等前向引用；脚注仅用于实际调用的外部素材；自助式解释不加脚注。
6. **法律解释方法贯穿全程**：在涵摄论证中须主动、显性地运用法律解释方法，受罪刑法定原则约束。遇到刑法分则条文解释时，优先调用 `methodology/interpretation-toolkit.md` 下编（含注意规定 vs 法律拟制鉴别、用语相对性、同位语测试、兜底条款三重过滤、罪刑均衡反向校验等专属 SOP）；总则用语解释及通用方法位阶则调用同文件上编。
7. **格式真相源**：`examples/canonical-report-template.md` 为报告格式的唯一权威基准。
8. **论证深度比例原则（proportionality）**：论证篇幅须与要件争议程度成正比——明显成立/不成立者一句带过，存在合理争议者展开正反论证，核心争议要件须穷尽方法论工具。
9. **法条全文展开**：每个被引用的法条第一次出现必须以引用块完整展示原文（至少到"款"级别）。
10. **客观第三人称**：避免"笔者认为"等主观色彩；当事人称谓全文统一。
11. **三维核验闭环**：增量法条校验、大纲覆盖+事实忠实、格式扫描——核验完成后按优先级一次性修订，不循环。

### 弹性层（可在标注后调整）

12. **报告字数区间**：默认总字数 ≤ 30,000 字；案情简单时可压缩至 8,000–15,000 字。
13. **反对说展开深度**：默认对每个核心争议要件展开正反两面论证；案情明朗时可一句话概括反对说。
14. **学说争议的立场选择**：当不同学说导致相同结论时，不展开学说争议（"结论一致则不问理由"原则）。
15. **排除罪名的表达方式**：对预选阶段排除的罪名，默认以脚注简述排除理由。

## 七、输出格式

**格式真相源**：`examples/canonical-report-template.md`。

**格式辅助规范**：`format/output-format-spec.md`。两者冲突时以前者为准。

要点摘要：

- A4 / 上下 2.54cm / 左 3.17cm / 右 3.07cm / **行距 1.25 倍**；
- 正文**宋体**小四（12pt）；部分/一级 黑体小三（15pt）/ 二级 黑体四号（14pt）/ 三级黑体小四（12pt）；引用块楷体小四（12pt）带左侧灰线；
- 段首 `  `（U+3000 全角空格 ×2）；
- 来源标识符四种：`[评注] / [案例] / [联网] / [KB:]`；
- **必有**目录页 + 大纲 + 正文双 H2 外壳层；
- **部分层条件化**：参与人数 ≥ 2 加 `# 第X部分`，= 1 跳过；
- 总字数 ≤ 30,000 字（含脚注）。

## 九、执行入口

收到触发请求时按下列顺序处理：

1. **检测 `.user-config.md`** 是否存在：
   - 不存在 → 执行 `workflow/0-cold-start-interview.md`，访谈完成后保存配置；
   - 存在 → 跳过冷启动，进入第 2 步。

2. **断点检测**：检查工作目录中是否存在 `progress.json`：
   - 存在 → 读取进度文件，从上次中断的阶段和写作单元继续执行，不从头开始；
   - 不存在 → 正常流程，进入第 3 步。

3. **解析用户输入**：
   - 用户已提交案件材料 → 进入 `workflow/1-case-intake.md`；
   - 用户仅询问方法论 / 单段调用 → 直接读取相应方法论文件回应。

4. **按工作流逐段推进**：W1 → W2 闸一 → W2 闸二 → W3 → W4。

5. **每段进入下一段前必须用户独立确认**（除非用户在冷启动配置中显式启用"全自动模式"）；W2 两闸**禁合并**。

6. **进度记录**：每个阶段完成或写作单元完成时，更新 `progress.json`：
   ```json
   {
     "phase": "W3",
     "writing_units": {
       "WU-1": {"status": "completed", "file": "draft-甲.md"},
       "WU-2": {"status": "in_progress", "file": null},
       "WU-3": {"status": "pending", "file": null}
     },
     "saved_artifacts": ["case-summary.md", "outline.md", "verification-log.md"],
     "last_updated": "2026-06-15T22:30:00+08:00"
   }
   ```
