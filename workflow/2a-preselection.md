# W2 闸一：候选罪名清单与独立确认

> 本文件定义工作流第二阶段的第一道闸门：基于 W1 事实单元，
> 按参与人逐一检索候选罪名，经双层校验后输出清单，独立确认。

| 禁止内容 | 说明 |

## 二、常用法条预加载

**目的**：刑法总则中若干条款在几乎所有刑事案件中都会涉及，在 Gate 1 启动前自动预校验，避免 Gate 2 追加校验的开销。

**预校验清单**（Gate 1 前自动执行）：

| 条款 | 内容 | 适用场景 |
|------|------|----------|
| §20 | 正当防卫 | 任何涉及防卫行为的案件 |
| §21 | 紧急避险 | 任何涉及避险行为的案件 |
| §22 | 预备犯 | 存在犯罪预备行为时 |
| §23 | 未遂犯 | 存在未遂形态时 |
| §24 | 中止犯 | 存在中止行为时 |
| §25 | 共同犯罪 | 存在多人参与时 |
| §26 | 主犯 | 存在共犯关系时 |
| §27 | 从犯 | 存在共犯关系时 |
| §28 | 胁从犯 | 存在被胁迫参与时 |
| §29 | 教唆犯 | 存在教唆行为时 |

**执行方式**：对清单中全部 10 条统一调用 `${LEGAL_SEARCH.get_article}` 批量校验，结果写入 `verification-log.md` 并标注 `[预校验]`。后续 Gate 1/Gate 2 中遇到这些条款时直接引用预校验结果，无需重复调用。

## 三、行为单元穷举预扫描

**目的**：在进入参与人导向的罪名检索之前，先执行一轮独立的"行为单元穷举扫描"，防止同一参与人的次要罪名被其主要罪名遮蔽。

**输入**：W1 输出的 case-summary.md 中的事实单元（§5）。

**执行步骤**：

对每个事实单元逐一执行以下扫描：

1. **提取参与人**：从该事实单元中提取所有涉及参与人。
2. **独立罪名检视**：对每个参与人，独立提问——"该参与人在此单元中的行为，还可能触犯哪些罪名？"
3. **全类别扫描**：使用 offense-examination-engine 的分类索引（财产/人身/公共安全/经济/职务/妨害司法）进行全类别扫描，不局限于最直观的罪名方向。
4. **输出罪名候选矩阵**：

```
行为单元 X → 参与人 A → 候选罪名 [a, b, c]
行为单元 X → 参与人 B → 候选罪名 [d, e]
行为单元 Y → 参与人 A → 候选罪名 [f]
...
```

**关键规则**：

- 一个行为单元可以同时涉及多个参与人的多个罪名
- 每个参与人在每个行为单元中的行为必须独立评价
- **不得因"该参与人已有主要罪名"而跳过次要罪名的检视**
- 穷举扫描结果与后续 §五 的参与人导向检索结果合并，去重后形成完整候选清单

## 四、情节节点检测

**目的**：在行为单元穷举预扫描之后、罪名检索执行之前，扫描时间线中的"转折型"情节节点，触发罪名升级检视，防止"知情后继续行为"等模式被遗漏。

**输入**：W1 输出的 case-summary.md 中的时间线（§2）和行为单元（§5）。

**检测规则**：

扫描时间线和行为单元，定位以下"转折型"情节节点：

| 节点类型 | 触发关键词 | 罪名升级检视清单 |
|----------|-----------|-----------------|
| 知情状态转变 | "得知……后""知情后仍""获悉……继续" | □ §114/§115 以危险方法危害公共安全罪 □ §161 违规不披露重要信息罪 □ §140 生产、销售伪劣产品罪（法条竞合检查） □ 不作为的故意伤害/杀人（保证人地位→召回义务） |
| 外部压力升级 | "投诉增多""调查介入""鉴定结论""监管通知" | □ 妨害作证罪（§307） □ 帮助毁灭、伪造证据罪（§307 第 2 款） □ 行贿相关罪名 |
| 身份状态改变 | "被停职""被调离""被免职""被解聘" | □ 职务侵占/挪用（身份丧失后的行为） □ 报复陷害相关罪名 |
| 责任能力波动 | "酒后""醉酒""情绪激动""精神状态异常" | □ 责任能力减等（§18 第 4 款） □ 原因自由行为（actio libera in causa）检视 |

**执行方式**：

1. 逐一比对上述节点类型与 W1 时间线中的 `★` 标记节点
2. 对检测到的每个节点，将对应的"罪名升级检视清单"中的罪名追加到 §三 行为单元穷举预扫描的候选矩阵中
3. 在 verification-log.md 中标注 `[情节节点触发]` 字样，记录触发的节点类型和追加的候选罪名

## 五、罪名检索执行

### 5.1 检索单位

按 **参与人 × 事实单元** 矩阵逐格执行：

```
参与人甲 × 事实单元1 → 候选罪名集 A1
参与人甲 × 事实单元2 → 候选罪名集 A2
参与人乙 × 事实单元1 → 候选罪名集 B1
...
```

### 5.2 法益导向法 + 法律检索 MCP 决策树

按 `offense-examination-engine/01-charge-discovery.md` 执行（含§五 MCP 法律检索协议——5 项能力槽、决策树情景 A-E、四级过滤铁律、条号入参格式）。同时参照 `offense-examination-engine/08-omission-check.md`（不作为犯罪检视触发）和 `offense-examination-engine/OVERVIEW.md §五`（非典型手段扩展匹配）进行补充扫描。

**Step 1：法益识别**
- 从事实单元核心行为出发
- 识别受侵害法益类型：
  - 个人法益：生命、身体、自由、财产、名誉、性自主权
  - 社会法益：公共安全、经济秩序、司法秩序
  - 国家法益：国家安全、公务行为
- 同一行为可能侵害多重法益 → 多路径检索

**Step 2：规范定位与分类**
- 依法益类型定位刑法分则章节（十章法益映射见 engine/01 §一）
- 在对应章节内匹配具体罪名
- 记录：罪名 + 条文编号 + 法益归属

**Step 3：法律检索 MCP 调用（按情景择路）**
- 情景 A（已知罪名+条号）→ `${LEGAL_SEARCH.get_article}`
- 情景 B（已知罪名找司法解释/数额）→ `${LEGAL_SEARCH.search_article}`
- 情景 C（仅有事实关键词）→ `${LEGAL_SEARCH.search_article}` 关键动词检索 → 看返回标题命中罪名 → 切回情景 A
- 情景 D（找类案）→ `${LEGAL_SEARCH.search_case}`
- 情景 E（行政犯前置法）→ `${LEGAL_SEARCH.get_law_list}`

### 5.3 法律检索 MCP 双层校验

每一个候选罪名必须通过以下双层校验，全部用法律检索 MCP 能力槽实操：

#### 层一：形式校验 —— 取条文原文 + 时效核验

调用模板：

```
qw_mcp_call:
  toolName: ${LEGAL_SEARCH.get_article}    # 由 .user-config.md legal_search.tools.get_article 解析
  arguments:
    title: "中华人民共和国刑法"               # 按 ${title_format} 契约
    number: "<按 ${number_format} 契约的条号>"  # 默认中文数字，如 "第二百六十四条"；"第133条之一"写作 "第一百三十三条之一"
```

返回字段判定：
1. `article` 非空 → 条号存在
2. `${timeliness_field}` 等于 `${timeliness_active_value}` → 通过形式校验
3. `${timeliness_field}` 不等于 `${timeliness_active_value}` → **拒绝引用**，整条候选回退；改用其他条号
4. `title` 若服务商自动追最新修正版（如返回"中华人民共和国刑法(2023修正)"）→ 在 verification-log 记录修正版本

#### 层二：实质校验 —— 司法解释/构成要件细化

调用模板：

```
qw_mcp_call:
  toolName: ${LEGAL_SEARCH.search_article}
  arguments:
    text: "<罪名> <要素关键词>"     # 例：" 盗窃罪 数额较大"、"诈骗罪 处分意识"
```

返回列表四级过滤（铁律，见 engine/01 §5.4）：

1. **时效**：剔除 `${timeliness_field}` 不等于 `${timeliness_active_value}` 的条目
2. **效力**：按 `${effectiveness_field}` 排序——`司法解释` ＞ `司法解释性质文件` ＞ `地方司法文件`
3. **主体**：含"最高人民法院"/"最高人民检察院"的 `issue_department` 优先
4. **时序**：同级取 `issue_date` 最新

实质校验目标：
1. 将条文构成要件要素 + 司法解释细化标准与案情事实初步对照
2. 确认罪名在三阶层体系中的位置：
   - 基本犯 / 加重犯 / 减轻犯
   - 行为犯 / 结果犯
   - 故意犯 / 过失犯
3. 初步判断：事实要素能否"触及"该罪名（非认定，仅筛选）

**筛选标准**：只要事实要素有可能满足任一客观构成要件要素，即保留为候选。
宁可多选，不可遗漏。

- 条文：刑法第[X]条第[Y]款

  - title 返回：[如"中华人民共和国刑法(2023修正)"]
  - ${timeliness_field}：[现行有效 / 废止或失效（取决于服务商命名）]
  - gid / url：[结果唯一标识 + 链接]

- 实质校验（${LEGAL_SEARCH.search_article}）：
  - query text：[实际入参]

- 法条竞合：特别关系/补充关系/吸收关系
- 共犯从属：从属于他人正犯行为

## 八、闸一确认

### AskUserQuestion 参数：

- question: "候选罪名清单是否准确？请确认或修改。"
- header: "闸一确认"
- multiSelect: false
- options:
  1. label: "确认无误" / description: "候选罪名清单完整准确，进入闸二（检视大纲生成）"
  2. label: "需要增加罪名" / description: "清单遗漏了应当检视的罪名，请补充"
  3. label: "需要删除罪名" / description: "清单包含明显不相关的罪名，请指出"
  4. label: "需要补充事实" / description: "案情信息不足以判断，需返回 W1 补充"

### 确认规则：
- **仅问一个问题**：候选罪名清单是否准确
- **禁止**在此处询问：
  - 大纲结构偏好（属于闸二）
  - 结论方向（属于 W3）
  - 知识库激活（属于闸二衍生问题）
- 用户选择"需要补充事实"→ 触发 W1 回环机制

## 九、违例自检

### 以下 4 项指标表明 AI 混淆了闸一与闸二内容：

| 编号 | 违例指标 | 检测方法 |
|------|----------|----------|
| V1 | 输出中出现"构成要件符合性/违法性/有责性"三阶层标题 | 文本扫描标题层级 |
| V2 | 输出中出现"争议焦点"、"核心争议"等评价语 | 关键词匹配 |

### 自检执行时机：
- 输出生成完毕、呈现给用户之前
- 若触发任一指标 → 自动回退重新生成
- 回退次数上限：2 次；超过则报告方法论困难，请求用户指导

## 十、特殊情形处理

### 10.1 未遂与预备

- 事实单元中行为未达既遂 → 同时列出：
  - 既遂形态候选（标注 [未遂检视]）
  - 对应预备罪（若刑法明确规定）

### 10.2 共犯形态

- 对从属参与人，罪名依附于正犯行为
- 清单中标注共犯形态可能性：
  - 教唆犯（《刑法》第29条）
  - 从犯（含"次要作用"与"辅助作用"两类，《刑法》第27条）
  - 共同正犯（对应《刑法》第25-26条规定的共同犯罪与主犯）
  - 胁从犯（被胁迫参加犯罪，《刑法》第28条）

### 10.3 数罪并罚预判

- 同一参与人跨事实单元的多个罪名 → 在"关系"列标注竞合类型
- 不在此阶段判断最终竞合结论（属于 W3 范畴）
