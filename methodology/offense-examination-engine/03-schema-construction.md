# Step 3：检视程式构建

## 概述

检视程式是鉴定式案例研习的骨架。它不是简单的清单或平面列表，而是以三阶层犯罪论体系为基础的逻辑树状结构。本步骤的任务是：基于Step 2的规范属性识别结果，为每一个候选罪名构建完整的、个案化的检视程式。

---

## 一、三阶层树状结构

### 1.1 结构总览

```
罪名检视程式
├── 第一阶层：构成要件该当性
│   ├── 客观构成要件
│   │   ├── 行为主体
│   │   ├── 行为客体/对象
│   │   ├── 实行行为
│   │   ├── 结果
│   │   ├── 因果关系
│   │   ├── 客观归责
│   │   └── 其他客观要素（情节、数额等）
│   └── 主观构成要件
│       ├── 故意 / 过失
│       └── 特殊主观要素（目的/动机等）
├── 第二阶层：违法性
│   ├── 违法性推定
│   ├── 正当防卫（第20条）
│   ├── 紧急避险（第21条）
│   └── 其他正当化事由
└── 第三阶层：有责性
    ├── 责任能力
    ├── 违法性认识可能性
    ├── 期待可能性
    └── 特殊免责事由
```

### 1.2 逻辑关系

- **阶层间**：递进关系。第一阶层否定则无需进入第二阶层；第二阶层否定则无需进入第三阶层。
- **阶层内**：客观要素与主观要素之间为连接关系（合取），全部满足方可进入下一阶层。
- **正当化事由间**：择一关系（析取），任一成立即阻却违法性。

### 1.3 树状结构的优势

1. **逻辑自洽**：清晰的检视顺序，避免论证循环
2. **经济性**：否定性结论可提前终止检视
3. **全面性**：确保每一层次的每一要素均被覆盖
4. **可扩展性**：可根据个案需要展开或收缩特定节点

## 二、构建算法（四子步骤）

### Step 3.0：模板匹配

**目标**：从通用检视程式模板库（`06-general-framework.md`）中选取与候选罪名最匹配的基础模板。

**双层模板库**（注意调用顺序）：

| 顺位 | 文件 | 用途 |
|------|------|------|
| 1（粗骨架） | `06-general-framework.md` | 通用检视程式骨架库（9 类犯罪通用模板）——**先在此选基本骨架** |
| 2（细程式） | `../crime-type-library/` 下 7 类型文件 | 针对故意作为犯/过失作为犯/不作为犯/共同犯罪/结果加重犯/认识错误/罪数与竞合的精细化检视程式——**对应类型存在时叠加细化** |

调用规则：先按 06 选基本骨架 → 再按罪名所涉犯罪类型在 `crime-type-library/` 中查找对应文件挂载（如本案为不作为犯叠加 `omission.md` 的保证人/作为可能性/等价性节点；涉共同犯罪叠加 `joint-crime.md` 的对应参与类型节点）。两库立场一致，crime-type-library 是 06 的精化版本。

**操作流程**：

1. 确定罪名的犯罪类型分类：
   - 结果犯 vs 行为犯
   - 具体危险犯 vs 抽象危险犯
   - 侵害犯 vs 危险犯
   - 作为犯 vs 不作为犯
   - 故意犯 vs 过失犯

2. 确定参与形态（注：中国《刑法》将参与人分为主犯、从犯、胁从犯、教唆犯，未明文区分共同正犯/间接正犯/帮助犯——下列为学理分析工具，最终归责仍须回到中国法的法定四类参与人）：
   - 直接正犯
   - 间接正犯
   - 共同正犯
   - 教唆犯（《刑法》第29条）
   - 从犯辅助作用类型（学理上的帮助犯，对应《刑法》第27条"起辅助作用"的从犯）

3. 确定完成形态：
   - 既遂
   - 未遂
   - 预备
   - 中止

4. 根据以上分类，从模板库中选取对应模板：
   - 如："结果犯 + 直接正犯 + 既遂" → 使用标准结果犯模板
   - 如："结果犯 + 共同正犯 + 未遂" → 使用共同正犯未遂犯复合模板

### Step 3.1：个案化调整

**目标**：将通用模板调整为适合本案的具体检视程式。

#### （一）子模板选择

根据具体罪名的特殊性，选择对应的子模板：
- 财产犯罪：加入"占有/所有"双层分析子模板
- 身份犯：加入"主体身份要素"前置检视子模板
- 目的犯：加入"超过的内心倾向"子模板

#### （二）裁剪无关节点

对于本案明显不涉及的要素，将相应节点标记为"略"：
- 如：行为人为成年人，精神正常 → 责任能力节点标记为"明显肯定，简述即可"
- 如：无正当防卫的事实基础 → 第二阶层正当防卫节点标记为"无需检视"

#### （三）扩展特殊规范节点

对于本案涉及的特殊规范，增加对应的检视节点：
- 如：涉及共同犯罪 → 在第一阶层增加"共同实行行为"、"共同故意"节点
- 如：涉及结果加重犯 → 增加"加重结果"、"加重结果与基本行为间的因果关系"节点
- 如：涉及不作为 → 增加"保证人地位"、"作为可能性"、"等价性"节点

#### （四）MCP条文具体化

利用法律检索MCP获取具体罪名的：
- 司法解释对构成要件的细化规定
- 指导性案例对不确定概念的具体化
- 学说对争议要素的解释方案

将检索结果嵌入对应节点作为检视参考。

---

### Step 3.2：无模板情形下的要素确定

当模板库中不存在对应模板时（如新型犯罪、生僻罪名），需从头构建检视程式。

#### 操作流程：

**第一步：条文解析**
- 提取条文中的全部构成要件要素
- 识别显性要素（条文明确规定）与隐性要素（需解释补充）

**第二步：构成要件要素提取**
- 主体要素：一般主体 / 特殊主体
- 行为要素：行为方式、行为对象
- 结果要素：危害结果、危险状态
- 因果要素：因果关系类型
- 主观要素：故意类型、特殊目的
- 情节要素：数额、情节加重/减轻条件

**第三步：学说观点参照**
- 检索刑法学教科书对该罪名构成要件的论述
- 对比不同学说对要素解释的差异
- 标注主要争议点

**第四步：检视程式成型**
- 将提取的要素按三阶层体系排列
- 确定要素间的逻辑关系
- 设定检视顺序

---

### Step 3.3：辅助规范挂载

**目标**：将辅助规范（定义规范、司法解释、量刑规范）挂载至检视程式的对应节点。

#### 挂载规则：

1. **定义规范挂载**
   - 条文中含不确定概念（如"国家工作人员"、"公共财物"）→ 在该要素节点下挂载对应定义条文
   - 示例：检视贪污罪"国家工作人员"要素时 → 挂载第93条

2. **司法解释挂载**
   - 对应要素存在司法解释具体化规定 → 挂载于该要素节点
   - 示例：检视盗窃罪"数额较大"时 → 挂载相关司法解释数额标准

3. **量刑规范挂载**
   - 在三阶层检视完成后设置"法律效果"节点
   - 依次挂载：法定刑档次 → 量刑情节 → 数罪并罚规则

4. **指导性案例挂载**
   - 对争议要素存在指导性案例 → 以注释形式挂载

---

### Step 3.4：防御规范跨阶层挂载

**目标**：将正当化规范和免责规范分别挂载至第二、第三阶层。

#### 操作规则：

**第二阶层挂载（正当化规范）**：
1. 检视案件事实中是否存在正当化事由的事实基础
2. 有事实基础的正当化事由 → 完整展开其检视子程式
3. 无事实基础的正当化事由 → 标注"无事实基础，不予检视"
4. 标准检视顺序：正当防卫（第20条）→ 紧急避险（第21条）→ 其他超法规正当化事由

**第三阶层挂载（免责规范）**：
1. 检视行为人是否存在免责事由的事实基础
2. 有事实基础 → 展开检视子程式
3. 标准检视顺序：责任能力（第17-18条）→ 违法性认识可能性 → 期待可能性

## 三、完整性检查清单

检视程式构建完成后，须逐项确认以下10项标准：

| 编号 | 检查项 | 确认 |
|------|--------|------|
| 1 | 第一阶层是否涵盖所有客观构成要件要素 | [ ] |
| 2 | 第一阶层是否涵盖所有主观构成要件要素 | [ ] |
| 3 | 因果关系与客观归责是否分设（结果犯） | [ ] |
| 4 | 共犯修正是否正确嵌入（共同犯罪情形） | [ ] |
| 5 | 未遂/中止修正是否正确嵌入（非既遂情形） | [ ] |
| 6 | 第二阶层是否列出所有有事实基础的正当化事由 | [ ] |
| 7 | 第三阶层是否包含责任能力、违法性认识、期待可能性 | [ ] |
| 8 | 辅助规范是否已挂载至对应要素节点 | [ ] |
| 9 | 法律拟制条款是否已设为独立检视路径 | [ ] |
| 10 | 检视顺序是否符合逻辑递进关系 | [ ] |

## 四、可扩展性设计

### 4.1 基础版

仅包含标准三阶层框架，适用于简单案件或明显成立/不成立的罪名。

### 4.2 展开版

在基础版上增加：
- 争议要素的学说分支
- 备选解释路径
- 类案参照
适用于核心争议罪名。

### 4.3 定制版

根据具体需求进一步扩展：
- 增加实务观点对比
- 增加政策考量分析
- 增加比较法参照
适用于疑难案件或学术研究。

## 六、与其他步骤的衔接

- **输入来源**：Step 2（规范属性识别）的规范分类结果、`06-general-framework.md`中的通用模板。
- **输出去向**：Step 4（要件拆解）将对检视程式中的每一要素节点进行精细化拆解。
- **迭代关系**：在Step 4拆解过程中如发现检视程式结构需要调整，应回溯本步骤进行修正。
